# Karden — Kubernetes Native Secret Lifecycle Manager

## Project Overview

A Kubernetes-native secret lifecycle management platform.
Developers attach annotations to their workloads, and karden automatically generates, injects, rotates, and audits secret values.

---

## Core Design Principles

- **K8s Secret as SSoT (Single Source of Truth)** — secret values are stored in etcd (K8s Secret); SQLite holds metadata only
- **Respect container immutability** — no Linux account password changes, no exec-based mutations
- **Follow K8s conventions** — developers write their own envFrom/secretRef; karden never breaks K8s standards
- **No black boxes** — every layer is understood and implemented explicitly
- **Interface-driven extensibility** — SecretStore and Rotator interfaces allow future integration with Vault, AWS Secrets Manager, etc.

---

## Architecture

```text
Developer (attaches annotations)
        │
        ▼
Watcher (Controller)
├── Detects Pod annotations in real-time via K8s Informer (not polling)
├── Automatically manages scaled-out Pod inventory
└── Triggers karden Server

karden Server
├── Password generation / rotation
├── K8s Secret creation and update
├── Built-in DB drivers (direct TCP connection, no exec)
├── Pod rolling restart trigger
├── Rotation scheduler
└── Audit logging

Web UI
├── Managed secret inventory
├── Manual rotation trigger
├── Manual secret registration (external API keys, etc.)
└── Audit log viewer

Storage
├── etcd (K8s Secret) — actual secret values
└── SQLite — metadata, audit logs, rotation history
```

## Annotation Spec

### DB Pod Annotations

```yaml
annotations:
  karden.io/inject: "true" # opt-in to karden management
  karden.io/type: "database" # database | generated | external
  karden.io/secret-name: "mysql-secret" # name of the Secret karden will create
  karden.io/db-type: "mysql" # mysql | postgres | mongodb | redis
  karden.io/db-host: "mysql-svc.default.svc.cluster.local"
  karden.io/db-port: "3306" # optional, defaults per db-type
  karden.io/db-user: "app_user" # DB account to rotate
  karden.io/rotation-days: "30" # rotation interval (default: 30)
```

Secret content karden generates (MySQL example):

```yaml
data:
  MYSQL_ROOT_PASSWORD: <auto-generated>
  MYSQL_USER: app_user
  MYSQL_PASSWORD: <auto-generated>
  MYSQL_DATABASE: <value of karden.io/db-name annotation>
```

### DB Pod Startup Sequence (timing guarantee)

```text
1. Developer deploys a Deployment
   └── envFrom.secretRef is declared
   └── Pod stays Pending because Secret does not exist yet (default K8s behavior)

2. Watcher detects the annotation

3. karden Server creates the Secret automatically
   └── Generates MYSQL_ROOT_PASSWORD, MYSQL_PASSWORD, etc.

4. K8s detects the new Secret
   └── Pod starts automatically (no extra trigger needed)

5. MySQL reads env vars and initializes
   └── Sets root password
   └── Creates app_user account
```

Leverages K8s default behavior (Pod stays Pending without Secret) — no custom trigger logic required.

### App Pod — Standard K8s Pattern

```yaml
spec:
  containers:
    - name: my-app
      envFrom:
        - secretRef:
            name: mysql-secret # references the Secret karden created
```

App Pods carry no karden annotations. Standard envFrom is used as-is.
Mutating Webhook for automatic envFrom injection is intentionally avoided — keeps K8s standards intact and prevents karden lock-in.

## Secret Types (karden.io/type)

| Type      | Behavior                                                               | Example                          |
| --------- | ---------------------------------------------------------------------- | -------------------------------- |
| database  | Generate password → ALTER USER on DB → update Secret → rolling restart | MySQL, PostgreSQL, MongoDB creds |
| generated | Generate random value → update Secret → rolling restart                | JWT secret, encryption key       |
| external  | Admin enters value via Web UI → inject into Secret                     | Stripe API key, Slack webhook    |

---

## DB Rotation Flow

```text
1. Generate new password
2. Connect to DB via TCP (built-in driver, no exec)
   ALTER USER 'app_user'@'%' IDENTIFIED BY 'newpass'
3. On success → update K8s Secret
4. Trigger rolling restart on referencing Pods (zero downtime)

On failure → rollback (Secret unchanged) + write audit log
```

**Why no exec?**

- Distroless images may not have a mysql client
- Separates DB-layer auth (password) from K8s-layer auth (RBAC)
- Direct TCP connection is more reliable

## Component Responsibilities

### Watcher

- K8s Informer-based (List-Watch, DeltaFIFO queue)
- Detects Pods with `karden.io/inject: true` → upserts into SQLite inventory
- Auto-registers scaled-out Pods, marks terminated Pods as inactive
- **No password mutation logic** — inventory management only

### karden Server

- Owns all rotation logic
- Built-in DB drivers (go-sql-driver/mysql, lib/pq, mongo-driver)
- Storage abstracted via SecretStore interface
- Rotation abstracted via Rotator interface per secret type

## Core Interfaces

```go
// Storage abstraction — swappable with Vault, AWS Secrets Manager, etc.
type SecretStore interface {
    Get(ctx context.Context, key string) (string, error)
    Set(ctx context.Context, key, value string) error
    Delete(ctx context.Context, key string) error
}

// Rotation abstraction — one implementation per DB type / secret type
type Rotator interface {
    Rotate(ctx context.Context, target Target) error
}
```

## SQLite Schema (metadata)

```sql
-- managed secret inventory
managed_workloads
├── id
├── pod_name
├── namespace
├── secret_name
├── type (database | generated | external)
├── db_type (mysql | postgres | mongodb | redis | null)
├── db_host
├── db_port
├── rotation_days
├── last_rotated_at
├── status (active | inactive)
└── created_at

-- audit log
audit_logs
├── id
├── target_id
├── action (rotate | view | manual_set)
├── actor
├── result (success | failure)
├── reason
├── created_at
└── metadata (JSON)
```
